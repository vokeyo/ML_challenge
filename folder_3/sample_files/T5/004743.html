FPD pot insertion.


Dmitri,

The FPD pots are suspected of causing the quench on Saturday 12/22/01 at
13:02. I have examined this in more detail and cannot determine if the
pots were responsible for the quench. The pbar halo losses did increase
during the time the pots were inserted, but the increase was relatively
small.

It appears that the magnet that quenched first was a low beta quad on
the B1 side of CDF which would be consistent with increased pbar losses.
However, the level of losses in C:B0ALOS while the pots were in place is
similar to the level of losses at the start of the store. (See attached
plot.) This leads me to believe that the pots were not the cause of the
quench.

For now I think it is reasonable to assume that the pots were not the
cause of the quench and continue to gain more experience. But we should
also closely examine any quench that occurs while the pots are inserted
before we can conclusively rule out the pots as the culprit.

Given the uncertainty, it is probably wise to keep the pots out of the
beam during the start of stores. However, instead of only using the last
2 hours I would suggest that the pots could be moved in during the last
8 hours of the store. This is a more useful amount of time for pot
studies. Acting as run coordinator I would be comfortable with such an
arrangement.

Mike

shift results

I spent the shift working on the VETO counters.  After a couple of false
starts, I measured A(peak)  and rates, where A is the raw signal
amplitude as measured by a scope, for each of V1S, V2S, V1N, V2N as
a function of HV.  For the south(north) counters, I normalized the rates
by LM-S(LM-N).  I then plotted the ratios as a function of HV; the results
are in the SCR logbook.  I used the plots to set the HV to 1.5kV.  There
is not a real plateau curve, but one can see where the PMTs go into
saturation for the S counters.

Viewing EPICS variables

There are several ways to examine epics variables from any online Linux
node:

- go to any online Linux node
- setup onl_apps

1) to see numbers, you can run PP.py with configureation file (I've put
three of them into /online/config/fpd/epics/ directory), for example

PP.py /online/config/fpd/epics/pos_variables.pp       -   position variables
PP.py /online/config/fpd/epics/hom_variables.pp       -   home variables
PP.py /online/config/fpd/epics/smo_variables.pp       -   smoke variables

If you see red colors, it means that the folloving variable in not
accessible

2) you can create your own *.pp configuration file (list of epics variables
you are interested, the
complete list of fpd variables is in
/online/config/fpd/epics/all_variables.pp and run PP.py with that file

3) to see stripchart of any variable in real time, you can run

> StripTool

and then supply any epics variable name to look at (you can superimpose
several plots in one graph)

4) another way to see stripchart is (one variable per graph)

> PvPlot.py

  from d0ol00 you can do "d0ssh d0ol13".  Note that you can go *to*
  any Online node, but only *from* d0ol00 or d0ol13 as they are the
  only nodes with the d0fpd keytab files.

As long as your account doesn't mess with setting DISPLAY, the ssh
tunnel for X should be set.

proposed logic for Stand-Alone triggers

A while ago, I proposed modifying our present logic in the SCR to be able
to easily expand on what we have.  Presently, it is difficult to add a
trigger, because of the availibility of open channels in the NIM logic.
In addition, we are unable to use one of the ADCs in our CAMAC crate.
I propose to remove the camera equipment from the SCR, and add another
NIM crate there.

The idea is to build triggers, as D0 does, using AND/OR terms.  These
terms are defined in the first drawing, and-or-terms.ps.  A legend
explains some of the definitions; I think the others are self-explanitory.

The second drawing, triggers.ps, shows the actual logic for both elastic
triggers, a loose diffractive -- DL trigger, and a tight diffractive -- DT
trigger.  The "Trigger Select" allows us to choose separate triggers.

In the following figures, crates1.ps and crates2.ps, crates1 refers to the
existing 3 NIM crates in the left rack, and crates2 refers to the proposed
2 NIM crates in the right rack.  The drawings are the front view of the NIM
crates; the top label CnSm refers to C -> crate and S -> slot.  The second
label is the module type:
LFIFO -> linear Fan In/Fan Out (or Logic)
OD    -> octal discriminator
4LU   -> 4 channel logic unit
GDG   -> gate delay generator
The first 5 slots of C1 are linear Fan In/Fan Out modules.  One channel
will go to the "E" early OD, one to the "I" in-time OD, one to the "U"
un-timed OD, and one to the first ADC in the CAMAC crate.  The last four
modules in C1 are the AND/OR terms.

The first 3 slots of C2 remain as they are.  C2S6-C2S8 are the VETOs for
the elastic and diffractive triggers.  C2S9-C2S11 are the triggers.

C3 presently only has GDGs for the "U" signals to go to the TDCs.

C4 will be used to make a grand OR of all the triggers to create the
"Trigger Select".  The output of this will be gated by CAMAC BUSY.
Individual triggers will also arrive here and be gated by CAMAC BUSY
to create the signals for the trigger mask in the CAMAC latch module.
The last four slots of C4 generates the logic and prescaled signals to
operate the visual scalers.  I still need to fill this in the drawing.

C5 will then contain all of the visual scalers.

I have not yet thought about how to incorporate dipole triggers.  Earlier
into this effort, I looked at the logic modules that we have, and believe
that we are ok.  I should be able to get another NIM crate from Regan's
area.  I'm not certain about whether or not we have enough of the right
length cables.

DFE Equations

    In principle, we could change the x cut from .99 to .98 and
does not change anything in terms of resources. I will try to
explain.

    The point is that the real geometrical separation is due to the
t value (for a given t value, events ranging from 0.90 to 1.00 in x
are spread over a small region of the detector, a dozen pixels or
so). And we are taking advantadge of this fact to implement
our trigger strategy, namely, use the finest segmentation for t-low
(t<1.0 GeV) events and a coarser segmentation for t-high
(t>1.0 GeV) events, under the assumption that the latter are
wanted regardless of their x value.

    So, we are already using the finer segmentation and
looking in deepest detail into the t-low region. Thus changing
the specific cut value of x will only cause to regroup the
equations that define xi-low and xi-high trigger equations.

    On the other hand, going from 0.99 to 0.98 makes the
xi-low region 0.02 wide, what means approximately 4 pixels
or 4 fine segments wide. Then, we could start thinking about
using "medium_segments" 2 segments wide (2/3 of a fiber)
in this region. But this is just a first thought. Of course, we
would need to carefully evaluate if it is worthy or not.

    In short, my understand is that changing the x-cut from
0.99 to 0.98 (or whatever) does not imply in increasing the
complexity of equations, but just reordering.

    I am attaching to this message some scatter plots that I
prepared yesterday showing the points in the x-y plane
where allowed tracks (according to M. Martens' tables)
traverse the several detectors. There are plots for t-low
and t-high tracks and they have been useful for us in order
to gauge the size of equations for these two regions. Maybe
they can be useful for others too.

new results from data (prelim to discussion tomorrow)

We have begun to look at other aspects of the data such has hit
resolution and fiber efficiencies that I will talk about tomorrow during
the meeting.

As a prelim to hopefully get people thinking about the problem, I'm
sending out the procedure I've used to get these very initial results.

Hit Resolution:

For Hit Resolution, starting with ideal detector geometry, I get a
subsample of events that have 1 segment in each plane (U, X, V). I then
take the center of the U/V pixel and convert it into (x,y) coordinates.
I also turn the center of the x segment into an x coordinate and then
look at the difference between the x from u/v and the x from the x
segment. This is the plot I will show tomorrow. I'm not sure how good of
a method this is to determine resolution, but it might be the best we
have.

There is an odd shift in the mean that needs to be studied that might be
due to the difference between ideal geometry and the real detector
geometry that is another iteration of the procedure. Or there could be a
bug in the code. :)

Fiber Efficiency:

For fiber efficiency, I start out by looking at a subsample of events
that have valid segments in the U and V planes. I then see how many of
these also have valid X segments. This is very preliminary and I still
need to take into account dead channels, hot channels etc.

It anyone has any comments or question about these measurements, how we
should go about them or anything like that, please bring it to the
meeting tomorrow so we can discuss them.

Vacuum leak in the A2 (GENTIL) pot.

A vacuum leak has developed in the A2/GENTIL FPD pot region. Presently
there is no urgency to remove the pot but we should consider making a
repair during the 2 week shutdown.

Apparently the leak begin near October 8th, 2001 which coincided with
the start of a 6 week shutdown of the Tevatron. On Oct 8th the vacuum
reading on the ion pump attached to the A2 pot jumped from a steady
value of 8E-12 Torr to about 1E-10 Torr and has been steadly climbing to
its present value of about 2E-9 Torr. (See attached plot of the ion pump
vaccum reading over the past 52 weeks. The ion gauge in this region has
failed so we do not have a vacuum measurement from the ion gauge.)
During the period since October the titanium sublimation pumps in the
adjacent separator have had to be fired in order to keep the separator
vacuum at a decent level.

Several weeks ago the beam valves between the separators and the A2 pot
region were closed and the separator and pot vacuum were isolated.
During this test the vacuum in the separator region improved and the
vacuum in the pot region deteriorated confirming that the leak was
somewhere in the pot region. An initial leak check of the pot region,
which includes a BPM, found no leaks on any of the beampipe conflat
flanges, no leaks on the flanges connecting the pot arms to the pot
castle, and no leaks on the upper or inner pot tips.

Today we leaked checked the internal bellows of the upper arm and found
a leak. The small, worm gear view port window was removed and helium
sprayed into the internal section of the upper pot arm. The leak
detector showed a rise in the He pressure from 2E-8 Torr to 1E-6 Torr
giving a clear indication of a leak. This means that there is a leak in
the internal bellows or the welds connecting the bellows to the flanges.
We will test the bellows on the other three pot arms this afternoon.

It seems prudent at this point to avoid moving the A2 upper pot any
further until some sort of repair/replacement can be made during the May
shutdown. The fear is that a small crack has developed in the bellows or
the weld which could become worse with continued flexing of the bellows.
For now the vacuum in the adjacent electrostatic separators is fine, but
if the vacuum deteriorates it could lead to sparking of the separators.

During the May shutdown we will attempt some sort of repair to the pot.
One suggestion is to remove the upper pot arm, move the inner pot arm to
the upper position on the castle, and blank off the inner port of the
castle. This leaves us without a inner arm but restores our vertical
spectrometer. With the upper pot arm removed we could inspect, test, and
hopefuly repair the upper pot arm.

I will put in a work request to have this repair work done during the
May shutdown so we can get the help we need (i.e. Doug Allen and
company) to perform the vacuum repairs. Perhaps Helio and LNLS would
have some insight or advice for us on this problem. One thing I am
worried about is trying to safely remove a pot arm in the horizontal
position without it "dropping" and crusshing the bellows. We will have
to look into this operation very carefully to make sure it can be done
safely.

Q1 signals to time them in with the Q2 signals (they were all adjusted
to within 1-2ns).  Doing this, however, creates a larger gap between
Q2 and Q1 in the early signal, so I added 54ns of delay to the Q2
signals into the early octal discriminator.

Both the LM and Veto counters are correctly timed in to form a VETO
for the elastic triggers.  There remains a mystery as to why the EOR
signals (halo particles) are 26ns out of time wrt the ECLK, so we are
presently without EOR signals in the VETO.

I also timed in the TDC signals such that the Q2 signal arrive 27ns
after the Q1 signals.  Though I haven't yet figured out why our EOR
terms are out of time, we can now apply an offline TDC cut to remove
any halo particles.

Even without the EOR terms are presently not in the elastic triggers,
the narrowing of the CLK gates has dramatically improved our rates:
at D0 L=7.6E30 the rate for EL_AD-PU is 15.11Hz!

The explanation for this result is that when the Q1 and Q2 signals were
not in time upon entering the discriminator (they were ~27ns apart due
to TOF), we used wide signals (60ns) to get a good overlap in the
coincidence unit.  We used 70ns CLK and ECLK gates, which corresponds
to 3 RF buckets.  Since when filling the buckets there is some spill, as
well as DC beam (not in a bucket), we had significant out of time signals
which made acceptable coincidences.  Narrowing the CLK and ECLK gates to
18ns removes most of the overlapping particles.

Lastly, the MAPMT signals are now timed into the ADC gate.  So, we're
ready to take data.

There are still a few things to do, but these will be non-disruptive to
the trigger setup.  I will follow up in the next couple of days with

                   Folks,
  please, let me deliver a few more details on the counting
  rates updating problem.
      In order to have an opportunity to have estimates of
 the counting scaler rates with as low frequency as possible
 DAQ does not update scaler EPICS variable each relatively short
 time interval but rather waits when a statistically significant
 (some ~100) number will be accumulated by some low frequency
 counting scaler. This is reasonable since any measurement
 with appropriate accuracy of low frequency is a slow process.
 But now after Vladimir has implemented rate watcher the
 situation can develop in the following way:

 1. Having seen that counting rate is dangerously high Vladimir's
    program turns HV OFF automatically and pops up alarm window.
 2. Naturally, with HVs OFF  DAQ sees zero counts of the scaler
    and waits for statistics accumulated in order to update
    EPICS variable. Note, that the last updated value is really
    big!
 3. Whatcher in its turn will not permit to turn HVs ON since
    it sees a big frequency value because DAQ can wait for a long time to
    update it.

      To solve this contradiction I suggest to update EPICS variable
   anyway after some reasonable time interval, let's say after a minute
   or two. It will limit our possibility to measure extremely low
   frequencies ( below ~1Hz) but will prevent DAQ and FPDGui from
   clinching.

       Now a few words more about further developments.
    We see that some undesirable things can happen while data taking.
   For instance, some voltage can go down. In order to have better
   control and check over the quality of the data we should have
   all the information available in our records. For me the best
   way to do it is to provide each data buffer with the status
   of all our EPICS variables. At this point some changes
   are necessary to be done.
     The main change:  Structure of Data Buffer!
   In this case the EPICS record information will precede events since
  it is more natural first to make a decision about the quality of the
  events and only after that send them to histograms. I do not think
  that it will change a lot in unpacking codes. You'll just
  make standard unpacking starting with another inital pointer to
  the buffer.

high loss rates

One of the big issues confronting the FPD is the high loss
rates.  In typical stores these days the home position rates at the
beginning of the store is 100khz! The rates increase by about
1.5 as we insert the pots.  This is not the behavior expected
of typical halo and our early time hit information indicates
the "traditional" halo rates at the operating position are on
the few Khz level roughly in line with predictions of
Drozhdin+Mokhov. The losses are typically in the form of spray
that lights up many fibers--the average occupancy is 14/112
channels for events satisfying the elastic trigger.
I believe this indicates a per channel effective rate of 14khz
if the trig scint rate is 100khz.  This situation brings up
several issue:

1) reducing the rates
There is a halo task force organized by Mike Albrow that is
meeting to understand and try to improve the halo situation.
CDF silicon is suffering from the high losses, so this reduction
effort has a lot of clout.  Unfortunately since it is not
tradional halo, but rather comes from some combination of parasitic
collisions, pressure bumps (bad vacuum areas in A+F sectors), DC beam (beam
that has left the bucket), or other effects, traditional collimation
does not seem to be effective at reducing it.

2) radiation damage to detectors
I believe Helio did some calculations on this with expected operating
conditions--he may even have done this for 100khz rates.  These studies
should be repeated in light of the high rates. I do not believe this is
a big issue, but this is not based on solid numbers. I would like Helio
to take charge of this issue if possible. Perhaps Nikolai has comments
on nature of particles from this spray?

3) long term damage to PMT's
We desperately need some understanding of what kind of rates
we can tolerate and operate for several years both for Phillips
and Hamamatsu tubes.  We have been using 100khz as a rule of thumb
on whether or not to insert pots, but I don't know what motivated
this.  I have temporarily waived this restriction for next couple
weeks before shutdown, but need input on this issue.
I would like Gilvan to take charge of this issue if possible.

4) operating in high rate environment
The other issue is can we do the physics with high background rates.
My preliminary answer is yes.  100khz/2.5Mhz is only 4% of crossings.
Assuming that we can make an adequate trigger this is a small efficiency
loss to reject these crossings.  I would like Jorge Molina (with advice
from Mario, Jorge B. and Mike S.) to  implement a segment counting
algorithm that parallels what Mario suggests that we could do in L1 to
apply a multiplicity cut to keep trigger rates under control.  This
algorithm should be tested on current data.

Comments, suggestions, additional help are all welcome.

the previous input of b = 7.0 GeV**(-2), but the t-reconstructed
distribution, although a little larger than the previous one, still  doesn't
agree with the experimental t-distribution for elastic events. Geometrical
acceptances values are being calculated and should be presented at next
meeting.

    4) Data Analyses status - J. Barreto

        Mike Strang has performed data analyses, using his own code, in
order to reconstruct X,Y transverse positions at each pot locations, using
the same elastic data sample as the RECO code (Runs 74-188), and these
reconstructed values are plotted in file C) above. They are very similar
to the plots obtained from the RECO Code, and it seems clear that there is
an excess of hits on the positive X-side at both P1 and P2 detectors, as
it was seen by the RECO code. Also, there is a clear shift of X=+3mm on
the location of P2.

    5) AOB

        The issue of taking elastic events using D1-D2 spectrometers was
discussed. Since Molina's thesis deals with dN/dt, it is not clear if the
previous data sample obtained using AU-PD spectrometers will allow such
study, since the t-reconstructed distribution seems contaminated with
non-elastic events. Maybe, a new set of elastic data will be necessary to
provide Molina with a good data sample for his thesis.
